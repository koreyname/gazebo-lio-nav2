# 项目整合问题与解决记录

## 1. 文档目的

本文记录 `lio-sam_simulation_navigation` 在整合 Gazebo、LIO-SAM、点云转换、
二维地图生成、AMCL 和 Nav2 过程中出现的主要问题、根因、解决方案和验证方法。

当前工作区的实际运行环境为：

- Ubuntu
- ROS 2 Foxy
- Gazebo Classic
- LIO-SAM 优化包：`lio_sam_op`
- Nav2

> 注意：Gazebo Classic 的生命周期警告不是本项目功能故障。真正需要处理的是
> `Address already in use`、模型生成失败、TF 断开和 Nav2 节点未激活等错误。

## 2. 最终系统结构

项目最终拆分为两个互斥的运行模式。

### 2.1 建图模式

```text
Gazebo
  ├─ /pointclouds ─> point_deal ─> /pointcloud_out ─> LIO-SAM
  └─ /imu ──────────────────────────────────────────> LIO-SAM

LIO-SAM
  ├─ 实时里程计和 TF
  ├─ RViz 全局拼接点云
  └─ /lio_sam/save_map ─> GlobalMap.pcd
```

启动命令：

```bash
./mapping.sh
```

建图模式不启动 AMCL 和 Nav2，Gazebo 底盘插件也不发布导航使用的
`odom -> base_link` TF，避免与 LIO-SAM 的定位结果冲突。

### 2.2 导航模式

```text
Gazebo
  ├─ /odom + odom -> base_link
  ├─ robot_state_publisher: base_link -> lidar_link
  └─ /pointclouds ─> point_deal ─> /pointcloud_out
                                      │
                                      └─ pointcloud_to_laserscan ─> /scan

Nav2
  ├─ map_server: my_map.yaml
  ├─ AMCL: map -> odom
  ├─ planner_server
  ├─ controller_server
  └─ bt_navigator
```

启动命令：

```bash
./navigation.sh
```

导航模式不启动 LIO-SAM 建图节点。

## 3. 问题与解决过程

### 3.1 建图和导航被同时启动

**现象**

- 启动一个脚本后，LIO-SAM、AMCL 和 Nav2 同时运行。
- 建图坐标系和导航坐标系互相竞争。
- RViz 同时显示建图内容、全局路径和局部路径，界面混乱。

**根因**

原启动逻辑没有区分建图和导航的职责。LIO-SAM 与 AMCL 都可能参与全局定位，
同时启动还可能产生重复 TF 发布。

**解决**

- 新增 `mapping.sh`，只启动 Gazebo、点云预处理和 `lio_sam_op`。
- 新增 `navigation.sh`，只启动 Gazebo、点云转 LaserScan、AMCL 和 Nav2。
- 使用 `publish_odom` 参数控制 Gazebo 底盘里程计：
  - 建图：`publish_odom:=false`
  - 导航：`publish_odom:=true`
- 将公共启动和清理逻辑放入 `scripts/launch_common.sh`。

### 3.2 RViz 没有点云或画面空白

**现象**

- LIO-SAM RViz 一片空白。
- `Map (global)`、注册点云或原始雷达点云没有数据。
- 有时话题存在，但 LIO-SAM 不接收。

**根因**

- Gazebo 点云字段不满足 LIO-SAM 对 Velodyne 点云的要求。
- LIO-SAM 订阅话题与实际发布话题不一致。
- 重复的点云降采样节点改变了话题链路。
- RViz Fixed Frame、点云 frame 和 TF 没有统一。

**解决**

- `point_deal` 直接订阅 `/pointclouds`。
- 将 Gazebo 点云转换为带有 `ring` 和 `time` 字段的 `/pointcloud_out`。
- LIO-SAM 的 `pointCloudTopic` 设置为 `/pointcloud_out`。
- 使用 `lidar_link` 作为雷达坐标系。
- 建图启动链路中移除多余的 `pointcloud_downsampling`。
- RViz 使用 LIO-SAM 配套配置，并以 `map` 为建图全局坐标系。

### 3.3 全局地图过一会消失，像是没有拼接

**现象**

- RViz 只能看到车辆周围的局部点云。
- 车辆移动后旧点云逐渐消失。
- 看起来没有形成全局拼接地图。

**根因**

- 显示的是当前帧或注册局部点云，不是 LIO-SAM 的全局关键帧地图。
- 全局地图可视化搜索半径或降采样参数过小。
- 坐标系配置为 `odom` 时，用户误以为缺少 `map`。

**解决**

- 建图模式将 LIO-SAM 的 `odomFrame` 设置为 `map`。
- RViz 区分显示：
  - 原始点云
  - 注册点云
  - LIO-SAM 全局地图
- 将 `globalMapVisualizationSearchRadius` 设置为足够覆盖整个 Gazebo 世界。
- 调整全局地图关键帧密度和体素大小，使旧关键帧持续参与拼接显示。

### 3.4 点云运行一段时间后变乱或抖动

**现象**

- 坐标轴、点云和轨迹抖动。
- 运行时间越长，地图错层越明显。
- 粉色轨迹或位姿标记持续变化。

**根因**

- 原始 LIO-SAM 参数面向真实 Velodyne 和实车 IMU，并不完全适合 Gazebo。
- 仿真雷达线数、水平采样数、频率和量程与配置不一致。
- IMU 噪声、外参、关键帧阈值和体素滤波参数不匹配。
- 如果同时显示原始点云和注册点云，会把正常的局部运动误认为地图抖动。

**解决**

根据 Gazebo 传感器和世界尺寸调整 `src/LIO-SAM/config/params.yaml`：

- `N_SCAN: 16`
- `Horizon_SCAN: 640`
- 雷达有效范围约为 `0.2` 至 `30.0` 米
- 调整 IMU 噪声和随机游走参数
- 设置 IMU 到雷达的外参
- 使用更适合室内仿真的体素滤波尺寸
- 收紧平面运动约束和回环匹配阈值
- 调整关键帧添加距离、角度和全局地图显示参数

RViz 中坐标轴和轨迹标记本身会随实时估计产生小幅变化。判断算法是否异常应观察：

- 静止时位姿是否持续大幅漂移
- 同一墙面是否出现明显多层重影
- `/lio_sam/mapping/odometry` 是否突跳
- TF 是否有多个节点重复发布

### 3.5 Gazebo 无法启动或小车偶尔加载不出来

**现象**

```text
Unable to start server[bind: Address already in use]
There is probably another Gazebo process running
```

或者：

- Gazebo 世界出现，但没有小车。
- `spawn_entity.py` 一直等待 `/spawn_entity`。
- 传感器话题没有出现。

**根因**

- 旧的 `gzserver`、ROS launch 或 RViz 进程没有退出。
- Gazebo 的 11345 端口仍被占用。
- `spawn_entity.py` 启动早于 Gazebo factory 服务。
- 仅检查话题名称不能保证模型已经生成成功。

**解决**

- 所有模式启动前调用 `stop_project_processes`。
- 清理本项目残留的 Gazebo、LIO-SAM、Nav2、RViz 和旧 ROS 查询进程。
- 将模型生成动作延迟到 Gazebo 启动之后。
- 启动后依次等待：
  1. `/spawn_entity` 服务
  2. `/imu` 和 `/pointclouds` 话题
  3. `/get_model_list` 返回模型 `robot`
- 如果任一步超时，停止后续节点，避免出现“RViz 已打开但机器人不存在”的假启动。

### 3.6 RViz 不显示小车模型

**现象**

- 地图和点云正常，但没有 RobotModel。
- TF 或坐标轴存在，小车外观仍不显示。

**根因**

- `robot_state_publisher` 没有启动或没有收到正确的 `robot_description`。
- RViz 没有添加或启用 RobotModel。
- URDF 根坐标与导航配置不一致。
- Gazebo 模型没有真正生成。

**解决**

- `robot_bringup` 统一启动 `robot_state_publisher`。
- 使用 `robo_car.urdf.xacro` 生成 `robot_description`。
- 统一使用 `base_link` 作为机器人根坐标。
- 启动脚本确认 Gazebo 中存在 `robot` 后才继续。

### 3.7 无法导航，TF 显示两棵不相连的树

**现象**

```text
Could not find a connection between 'odom' and 'base_link'
Tf has two or more unconnected trees
Message Filter dropping message: frame 'lidar_link' ... reason 'Unknown'
```

RViz 的 Navigation 2 面板显示：

```text
Navigation: unknown
Localization: unknown
```

**根因**

Gazebo `libgazebo_ros_planar_move.so` 默认发布：

```text
odom -> base_footprint
```

但当前 URDF 没有 `base_footprint`，模型根坐标是 `base_link`。因此系统形成：

```text
odom -> base_footprint

base_link -> lidar_link
```

两棵独立 TF 树导致 AMCL、代价地图和 RViz 无法将 `/scan` 转换到导航坐标系。

**解决**

在 `robo_car.urdf.xacro` 的 Gazebo 底盘插件中明确配置：

```xml
<odometry_frame>odom</odometry_frame>
<robot_base_frame>base_link</robot_base_frame>
```

修复后导航 TF 链为：

```text
map -> odom -> base_link -> lidar_link
```

其中：

- AMCL 发布 `map -> odom`
- Gazebo 发布 `odom -> base_link`
- `robot_state_publisher` 发布 `base_link -> lidar_link`

### 3.8 `/scan` 不稳定或导航节点收不到雷达

**现象**

- `/pointcloud_out` 存在，但 `/scan` 没有有效数据。
- AMCL 或 costmap 丢弃 `lidar_link` 消息。

**根因**

- 点云转 LaserScan 的高度范围、频率和量程与 Gazebo 雷达不一致。
- 转换节点启动时 TF 尚未建立。

**解决**

调整 `pointcloud_to_laserscan_launch.py`：

- `target_frame: lidar_link`
- `transform_tolerance: 0.05`
- 高度范围：`-0.10` 至 `0.50` 米
- `scan_time: 0.125`，对应 8 Hz 雷达
- 量程：`0.12` 至 `30.0` 米

`navigation.sh` 在启动 Nav2 前等待 `/scan` 出现。

### 3.9 Nav2 只启动了一部分节点

**现象**

- `map_server` 和 AMCL 已启动。
- `controller_server`、`planner_server` 停留在 `inactive`。
- `bt_navigator` 退出或 `/navigate_to_pose` 没有 action server。

**根因**

当前系统是 ROS 2 Foxy，但行为树插件列表缺少 Foxy 默认行为树使用的：

```text
nav2_initial_pose_received_condition_bt_node
```

行为树初始化失败后，导航生命周期管理器无法完整激活所有节点。

**解决**

在 `robot.yaml` 的 `bt_navigator.plugin_lib_names` 中补充该插件。

最终应存在以下 action：

```text
/compute_path_to_pose
/follow_path
/navigate_to_pose
```

### 3.10 导航启动后没有 `map -> odom`

**现象**

- 地图已加载，但完整 TF 链尚未形成。
- 必须手动点击 RViz 的 `2D Pose Estimate` 后才能导航。

**根因**

AMCL 必须获得初始位姿后才发布 `map -> odom`。通过脚本向 `/initialpose`
只发布一次存在启动时序竞争，消息可能在 AMCL 激活前被发送。

**解决**

因为 Gazebo 中小车固定出生在 `(0, 0, 0)`，直接在 AMCL 参数中配置：

```yaml
set_initial_pose: true
always_reset_initial_pose: true
initial_pose:
  x: 0.0
  y: 0.0
  z: 0.0
  yaw: 0.0
```

这样 AMCL 激活时自动使用 Gazebo 出生点，不再依赖一次性话题。

如果以后修改了 Gazebo 出生位置，必须同步修改这里的初始位姿。

### 3.11 局部代价地图持续报告 `map` 不存在

**现象**

在 AMCL 完成初始化前反复出现：

```text
StaticLayer: "map" passed to lookupTransform argument target_frame does not exist
```

**根因**

局部代价地图使用 `odom` 作为全局坐标系，却错误加载了依赖 `map`
的静态地图层。静态地图层通常只需要放在全局代价地图。

**解决**

- 局部代价地图只保留 `voxel_layer` 和 `inflation_layer`。
- 全局代价地图保留 `static_layer`、`obstacle_layer` 和 `inflation_layer`。

### 3.12 三维点云如何转换为二维导航地图

**问题**

LIO-SAM 保存的是三维 `GlobalMap.pcd`，Nav2 需要的是二维占据栅格
`.pgm + .yaml`，二者不能直接互换使用。

**解决**

新增两级保存流程。

保存三维点云：

```bash
./save_map.sh
```

内部调用：

```text
/lio_sam/save_map
```

输出：

```text
maps/lio_sam_map/GlobalMap.pcd
```

保存并生成二维导航地图：

```bash
./save_2d_map.sh
```

脚本会：

1. 保存最新的 LIO-SAM 三维地图。
2. 按高度范围截取障碍物点。
3. 投影到 XY 平面。
4. 按分辨率生成占据栅格。
5. 执行最小点数过滤和适度膨胀。
6. 输出 Nav2 地图。
7. 重新安装 `robot_navigation2`。

输出：

```text
src/robot_navigation2/maps/my_map.pgm
src/robot_navigation2/maps/my_map.yaml
```

## 4. 最终使用流程

### 4.1 编译

```bash
source /opt/ros/foxy/setup.bash
colcon build --symlink-install
source install/setup.bash
```

### 4.2 建图

```bash
./mapping.sh
```

在 Gazebo 中控制小车遍历环境，确认 RViz 的全局地图持续拼接。

### 4.3 保存二维地图

```bash
./save_2d_map.sh
```

可选参数：

```bash
./save_2d_map.sh 0.05
```

其中 `0.05` 是栅格分辨率，单位为米/像素。

### 4.4 停止建图

```bash
./stop.sh
```

### 4.5 导航

```bash
./navigation.sh
```

启动完成后，RViz 中应显示：

- 二维地图
- 小车模型
- LaserScan
- AMCL 粒子或定位结果
- 全局和局部代价地图

然后使用 RViz 的 `Navigation2 Goal` 设置目标点。

## 5. 验证命令

### 5.1 检查小车是否生成

```bash
ros2 service call /get_model_list gazebo_msgs/srv/GetModelList '{}'
```

返回结果中应包含：

```text
'robot'
```

### 5.2 检查关键话题

```bash
ros2 topic list | grep -E '^/(imu|pointclouds|pointcloud_out|scan|odom|map)$'
```

### 5.3 检查 TF

导航模式：

```bash
ros2 run tf2_ros tf2_echo odom base_link
ros2 run tf2_ros tf2_echo map lidar_link
```

两条命令都应持续输出有效变换。

### 5.4 检查 Nav2 action

```bash
ros2 action list -t | grep -E 'navigate_to_pose|compute_path_to_pose|follow_path'
```

应包含：

```text
/navigate_to_pose
/compute_path_to_pose
/follow_path
```

### 5.5 检查生命周期

```bash
ros2 lifecycle get /map_server
ros2 lifecycle get /amcl
ros2 lifecycle get /controller_server
ros2 lifecycle get /planner_server
ros2 lifecycle get /bt_navigator
```

正常情况下均应为：

```text
active [3]
```

## 7. 后续修改注意事项

1. 修改 Gazebo 小车出生点时，同步修改 AMCL 的 `initial_pose`。
2. 修改雷达频率时，同步修改 `pointcloud_to_laserscan` 的 `scan_time`。
3. 修改雷达线数或水平采样数时，同步修改 LIO-SAM 的 `N_SCAN` 和
   `Horizon_SCAN`。
4. 不要同时运行 `mapping.sh` 和 `navigation.sh`。
5. 不要让多个节点同时发布同一段 TF。
6. 导航使用二维栅格地图，不直接使用 `GlobalMap.pcd`。
7. 如果再次出现 Gazebo 端口占用，先执行：

```bash
./stop.sh
```

8. 如果修改源码而运行结果没有变化，重新编译并重新 source：

```bash
colcon build --symlink-install
source install/setup.bash
```
